**4.1 Идентификатор прецедента:** A2

**4.2 Название прецедента:** Трансформация данных

**4.3 Контекст:** Интеграция данных

**4.4 Участники (actors) и цели (goals):**

| Участник  | Категория  | Цель (goal) |
|---|---|---|
| Инженер данных | Основной  | Подключять источники данных, формировать ETL конфигурацию |
| Аналитик | Внешний  | Формирвать метрики, формулы агреграций |
| База данных (источник) | Инструмент| Хранение не обработнных данных |
| База данных (хранилище) | Инструмент| Хранение обработнных данных |
| ETL/ELT | Инструмент| Извлечение, преобразование, загрузка данных|

**4.5 Предусловия (pre-conditions):**

* Условие наличия потока управления: BI, Требования к данным.

* Условие наличия входного потока: Поток данных.

* Условие наличия потока исполнителя: Инженер, Аналитик.

* Условие наличия потока механизма: Работающая система (результаты выгрузки).

**4.6 Постусловия (post-conditions):**

* Выходной поток: Конфигурация ETL задач.

**4.7 Основной поток выполнения (main flow)**:

| Участник  | Действие (activity)  | Ожидаемый результат |
|---|---|---|
| Инженер | Проектирование сисетмы интеграции | Конфигурация ETL |
| Аналитик | Оперделение проверочных мерик, формул агрегаций | Параметры проверки данных |
| ETL | Извлечение | получение первичных данных |
| ETL | Трансформация | преобразование первичных данных |
| ETL | Загрузка | получеине аналитических данных |

**4.8 Исключения (exceptions):**

| Условие (риск) | Последствия | Реакция |
|---|---|---|
| Невозможность подключения к источнику данных| Недоступность данных | Выполнение повтороного подключения |
| Получение данных неверного формата | Получение ошибочных значений | Выполнить конвертацию данных |
| Изменение архитектры источника данных | Ошибка сопоставления данных | Нет |

**4.9 Альтернативы (alternates):**

| Участник  | Действие (activity)  | Ожидаемый результат |
|---|---|---|
| Аналитик | Новый источник данных | Дополнение системы интеграции новым источником |

**4.10 Временные параметры:**

* **Триггер (событие, стартующее прецедент):** Задача по расписанию.

* **Номинальная частота повторения прецедента:** Раз в сутки.

* **Продолжительность прецедента:** 2 часа

### 9. Подготовка к интерпретации построенных моделей

**9.1 Используемые паттерны проектирования и разработки [✋](https://github.com/stankin/design-part-2/wiki/sem2):**

* **Процессная модель для сравнения:**
Задача: необходимо обеспечить поток аналитических данных

Решение задачи при помощи методологии PDCA:

1. Этап Plan(Планирование):
* Выявленные проблемы: отсутствие данных для проведения аналитических мероприятий
* Цель: сформировать аналитический поток данных
* Требования: поддержка актуальности получаемых данных
* Ожидаемый результат: автоматизированная задача выгрузки данных
* Ресурсы, необходимые для достижения ожидаемого результата: функционирующая информационная система источника данных

* Процессы (запланированные действия):
    - формирование потока данных
    - разработка ETL процессов
    - интеграция информационных систем

2. Этап Do(Выполнение):
Инженер данныз и аналитик выполняют поставленные задачи

3. Этап Check(Проверка):
По итогу внедрения разработанной системы производится оценка эффективности

* **Используемые в разработке паттерны и фреймворки:**

**9.2 Используемые паттерны выявления проблем [✋](https://github.com/stankin/design-part-2/wiki/sem3):**
* Муда: рост объема данных, приводящих к потребности дополнительных ресурсов
* Мура: изменение архитектуры источника данных или целевого хранилища в конце разработки ETL
* Мури: выбор сервера для ETL, которые не отвечают по своей мощности требуемым характеристикам (объем хранимых данных, производительность и т.д.)


**9.3 Возможные антипаттерны [✋](https://github.com/stankin/design-part-2/wiki/sem4):**

| Категория  | Антипаттерн (риск) | Действие по избежанию |
|---|---|---|
| Разработка | Изобретение колеса/велосипеда: Создание с нуля, вместо использования готового решения.| Изучить портреты готовых решений |
| Архитектура | Членовредительство: Излишнее «затачивание» объекта под определенную очень узкую задачу таким образом, что он не способен будет работать с никакими иными, пусть и очень схожими задачами. | разбиение процесса на подпроцессы |
| Организация | Расползание рамок: потеря контроля над разрастанием проекта. | распределние ответственности по проекту |
| Среда | Город ларьков — каждый отдел вырабатывает свой собственный механизм обмена информацией. | определение централизованный среды |

Базовое получение отчётности занимает порядка 10 мин на 1 таблицу, и порядка 35 мин на весь объект в целом (без учета загрузки в BI, что может занимать еще порядка 15 мин), в рабочее время
**10.3 Расчет потенциального эффекта от автоматизации:**
Получение отчётности путем организованного ETL процесса занимает порядка 34 сек на таблицу, и менее 2 мин на объект в целом, в не рабочее время.
Таким образом на выгрузку данных всего объекта затрачивается меньше времени более чем в 17 раз.
Помимо экономии времени, за счет автоматизации выгрузки, выполнение ее можно поставить в нерабочее время что дает возможность получения отчетности на начало рабочего дня, так же возможность сокращения частоты обновления.
